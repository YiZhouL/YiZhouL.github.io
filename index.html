Git远程操作

发表于 2020-11-08 分类于 Git

现在的项目几乎都是由多人合作完成维护的，通常会使用代码托管平台帮助团队管理、维护项目。

目前主流的平台有github、gitlab(可看做私人github)，这些平台同样使用git管理项目，而git提供了丰富的工具达到协作的目的。

认证

git需要和操作远程仓库前，要进行认证处理

git采用ssh连接，需要在github平台配置ssh公钥

操作核心

  1. 分支是本地与远程仓库交互的基本单位，任何与远程仓库有关的操作就是本地分支与远程分支的操作
  2. 只要把远程分支当做普通的本地分支操作即可
  3. 本质上是在本地维护远程分支

规范

  1. 团队协作的性质导致本地远程分支几乎不会是远程仓库中最新的，基本会滞后
  2. 本地远程分支不能直接提交(在工作目录修改后提交)，永远在自己的工作分支进行修改，然后合并到本地远程分支
  3. 依据1，将自己的工作合并到本地远程分支时，先从远程仓库更新本地远程分支
  4. 使用本地远程分支更新远程仓库对应分支

之所以这样做，是因为git更新本地远程分支和远程仓库分支的时候默认采用ff合并

按照规范操作永远不会发生错误和冲突，只是一个ff合并，并且分支是一条简明的直线。

手动关联远程仓库

其中fetch是git中的远程仓库引用及相关数据对象的下载器

remote会定义远程仓库的fetch设置，默认将会下载远程所有分支，标签默认不下载

fetch拉取的引用都会放在refs/remotes/name/*

同时每次会生成FETCH_HEAD文件记录fetch的相关引用和信息，每次fetch都默认会覆写FETCH_HEAD

要注意，fetch一个远程分支的时候并不会直接创建一个对应的本地分支，需要额外的参数

总结

git的命令有丰富的参数可以完成很多场景下的操作，并不简单的局限于功能描述信息。

我比较倾向于将命令的功能细分、模块化；但是那样会产生许多步骤，所以实际工作中还是要掌握常用参数的

例如switch本意是切换分支，但是也具有创建分支的功能，这些参数都是在实际使用中有价值的，而这两篇主要记录标准的流程下必知必会的git基本命令。以后实际工作运用时有用的命令再更新吧。

Git设置

git通过不同的设置层次级别实现定制化。

在git有四种设置级别：

  1. system：系统级别，配置文件位于 /etc/gitconfig
  2. user：用户级别，配置文件位于 ~/.gitconfig (旧版本位于~/git/config)
  3. repository: 仓库级别，配置文件位于 .git/config

一个git仓库最终的配置项将会从上到下不断更新(重复将覆盖)的结果。

基本工作流

git最基本的工作流如下：

  1. 创建一个仓库
  2. 修改或添加一些文件(位于工作目录)
  3. 把修改的的文件更新暂存区
  4. 将新的暂存区进行提交
  5. 重复2-4
更新暂存区

git使用git add命令更新暂存区，把git add看成git add change的缩写会更好。

change含义几乎包含文件的所有属性，其中一些属性需要在config中设置(如文件权限)。

提交版本

git commit (INDEX) 把当前的暂存区作为一个新版本提交到git仓库。

只有当INDEX存在，或者INDEX发生变化提交才会生效，当初始化一个仓库不做任何暂存时，INDEX为空即不存在。

回退操作

git推荐的工作模式是一点一点往仓库添加内容，删除是少数的；如果一切正确执行，工作流就像以上不断添加、不断提交

不过有时候也会由于人为失误或其他原因，需要中断去执行回退操作。

正常提交时数据流向为：WORKTREE -> INDEX -> COMMIT；因此可能发生回退的情况有以下三种

工作目录文件回退

当我们正在编辑某些文件但出现了一些错误想把文件回退到初始状态

git restore (files) 命令根据源数据恢复指定的文件

源数据默认是当前INDEX，也可以是commit；本质上源数据最终所使用的数据就是一个tree对象，INDEX是一个待提交的tree对象，commit包含了提交的tree对象。

restore将会从tree中查找file，如果存在就会用该文件回退；
如果不存在，将会检查file当前是否处于追踪状态，在追踪将会删除file，否则报错。

INDEX回退

restore也可以用来恢复INDEX(也是个文件)，常用在需要回退某些文件到上次暂存时状态的时候

当加上 –staged 参数时，只会回退INDEX中的特定文件；如果同时需要将工作目录中的文件也回退时，要加上 [-W|–worktree]

Notes

  1. 使用INDEX恢复时，一定要注意多次暂存后，就不能恢复到最初那几次暂存前的状态，比如暂存了1、2、3、4，就只能恢复到4。
  2. 当-s指定源时，推荐确保该文件的确存在，这样restore才有意义。
COMMIT回退

有时候也需要把错误提交进行回退，git使用reset命令完成。

git reset会将当前HEAD重新指向特定的commit对象，通过不同的reset级别会对INDEX和work tree做出不同行为。

reset工作过程为：

  1. 首先将HEAD 指向TARGET
  2. 如果是soft，保持working和INDEX不变；mixed则将INDEX指向commit所包含的TOP tree；hard会在mixed基础上检出INDEX作为working目录内容

分支合并

分支合并是团队开发中最频繁最重要的操作，也是问题出现最多的地方；需要了解git合并策略和冲突产生的规则。

git合并采用三方合并，git会寻找一个base commit，然后将当前分支A、待合并分支B三方进行合并，如果同一文件中A和B内容都不同，则产生冲突；对于父节点不唯一的会递归找出base commit。

git根据A、B分支不同的关系划分了几种不同的策略。

  1. A、B为父子关系

    当A合并B时，将会直接把A分支重新指向B的commit；当B合并A时无变化，因为是没有意义的。这种合并称为“**快进(fast-forward)**”

  2. A、B分支为兄弟关系

    A合并B时将会在A分支创建新的commit作为合并后的结果。

注意

  1. 合并时，保持A、B是clean的，当合并出现任何问题时可以保证数据不会丢失
  2. 合并时，禁用ff；永远手动解决冲突
  3. 当出现conflict时，使用git status可以查看冲突的文件

Git内部设计

发表于 2020-11-08 分类于 Git

git作为当前最流行的版本控制系统(CVS)不管是出于学习或工作目的都是非常值得学习的。本文将着重从git内部工作原理出发记录自己的学习心得，了解git的内部原理可以迅速帮助我们真正使用git，而不仅仅停留在命令表面。

版本控制系统

版本控制系统在平时学习中其实经常就打交道，就拿毕业论文来说，基本上我们每修改一次，都会复制然后改个文件名标记一下，这就是一次完整的版本控制过程。

CVS本质上就是做这样的工作，但是它提供了一种高效的方式。想一想，当你的项目文件（或者你想跟踪的任何文件）很少或修改并不频繁时，完全可以手动控制，但是一旦你的需要修改的文件非常多，那么手动管理就会变得很困难而且容易出错，而CVS使用很多方法保证了每一次所做的修改都能做到精确控制。

当作为一个开发者时，使用CVS是非常必要的。CVS可以帮助开发者了解项目的开发过程，了解每一个版本的修订历史，当出现问题时可以进行版本回溯；如今几乎所有的工程项目都是以团队为单位，而CVS的出现极大提高了团队协作开发效率、降低了管理难度。

git和gitHub关系

官方定义

Git is a free and open source distributed version control system designed to handle everything from small to very large projects with speed and efficiency.

GitHub is a development platform inspired by the way you work. From open source to business, you can host and review code, manage projects, and build software alongside 50 million developers.

Git帮助开发者管理项目，GitHub使用git帮助团队管理项目。

git特点

  • 速度（即高效）
  • 设计简单
  • 支持超大规模项目
  • 非线性开发模式（允许上千个并行开发的分支）
  • 完全分布式

文件存储

传统CVS的存储方式

大多cvs记录每个版本之间的差异，某一版本的完整文件由初始版本叠加这些差异即可得。

git的存储方式

git采用了简单的记录方式，当某个文件发生改变时，直接存储文件的快照（看作为新文件）而非差异

每个文件有唯一的索引（采用某些hash算法）

当某个新版本的文件没有改变时，并不会创建一个新文件，而是使用上个版本的index

只要知道某个版本的所有index即可获取该版本完整内容

git自下而上大量采用了index这种思想，git中的操作对象更像是index，而不是文件本身

标准工作流

git划分了以下三个文件区域

  • 工作目录：修改文件
  • 暂存区：保存工作目录中文件的最新索引
  • git仓库：将暂存区的索引作为一次新版本进行提交

因此，git中的文件具有三种状态，对应三个工作区域

  • 已修改
  • 已暂存
  • 已提交

git仓库的构建

blob对象

在git中，一个普通文件称为blob对象，具有唯一的index

tree对象

文件夹在git中表示为tree对象，文件夹是一个特殊的文件，包含其他文件的索引，本身也具有唯一index

暂存区

暂存区本质上是一个索引集，而索引集想一想是不是一个tree对象。因此要明白，提交一个新版本时，提交的是一个tree对象，该tree对象包含了工作目录中所有的文件，并把它们构造成一个tree对象，是一个TOP tree对象。

commit对象

提交一个tree对象的结果就是一个commit对象。

commit对象是一个特殊的文件，包含了提交tree对象的index和一些提交信息。

如果知道一个commit的index，那么就可以获取完整的文件内容。

扩展
  1. 暂存区并不是实时对应工作目录的最新内容，只有暂存后才会更新暂存区
  2. 当提交完暂存区后，暂存区并不会清空；暂存区的变化与提交无关
  3. 文件（或对象）的索引只与内容有关，与文件名无关
  4. 构建快照和生成索引发生在暂存过程中；验证数据的完整性（通过再次计算文件的索引判断和已有索引是否相同）发生在提交过程中
问题

  1. git如何感知文件名的变化

    git中的tree对象中不仅仅存储着包含的index，还包含对应文件的名字

    文件名的变化发生在所属的tree对象，这时候暂存时将会创建一个tree对象的快照

  2. git如何控制文件的权限

    同文件名一样，tree对象还包含每个文件的权限，当权限发生变化时，将会创建一个tree对象的快照；但是一般不对权限做控制，如果需要，要做对应的设置

  3. git如何回溯版本

    在commit中，还会保存上一个commit的index，通过这样简单的方式就可以将所有的commit链接起来，回溯就成了一个链表索引的动作

分支

git创建一个分支是非常简单的，几乎是瞬间完成，git中的分支模型可以称得上是“大杀器”。

引用

现在，我们只需要知道一个commit的index即可完全知道所有的历史提交，但是index是一串无意义的字符串(由sha256计算)，不好记忆且对于开发者来说没有意义。

引用实质上是一个commit的别名

在git中引用使用一个简单的文件存储，文件内容为commit index，而文件名为引用名。

分支的创建

分支是引用概念的延伸。随着项目版本不断迭代，每一次commit将会把这次的index链接到上次的commit index。

分支本质上是一个index会更新的引用，而不是某个commit index不变的引用。分支所指向的index永远是当前最新的commit index。

默认分支

在git中，总是存在一个默认分支，名为master（从2020/10/1更名为main，不过相关命令貌似还是master），当你没做过任何有关分支切换的操作，所有的修改、提交都在该分支。

多分支

从git来看，多分支是多个index可变的引用，所有这些分支都可以进行提交。

从项目管理来看，不同的分支流则代表不同的任务流；可能是新功能、BUG修复等等。

HEAD文件

当提交一个commit时，肯定是在当前工作的分支上进行的，git使用HEAD文件标记当前的工作分支。
如同引用是commit index别名一样，HEAD是分支的别名，内容和分支一样是可变的。

每一次分支切换时，HEAD所指向的分支则会改变。

Tag标签

tag如同名字一样，用来给git的commit对象打上标签。

git存在两种标签，轻量标签（lightweight tag）和附注标签（annotated tag）；这两种标签差别仅仅就在于实现方式不同，前者是后者的简单实现，因此功能比后者要少

轻量标签

项目总是会有很多版本，团队几乎都会对版本进行一个简单的版本号标记，在此情景下，打上轻量标签非常合适。

轻量标签是一个不变的引用，引用名即为版本号，非常简单的实现。

附注标签

有时候，团队可能不仅要标记一个版本号，还会对该版本添加一些额外的信息，可能是某些功能信息等等；这时候轻量标签则实现不了了。

附注标签创建了一个tag对象，该对象含有被打标签的commit index和额外信息。

轻量标签指向commit对象，附注标签指向tag对象，这就是所有的不同，轻量标签完全可以用附注标签代替，但没有必要。

需要注意的是附注标签中的index可以不是commit index，也可以是blob对象、tree对象。

git中的数据对象

从以上两部分可以知道，git中有四种数据对象，实际上所有的git操作都是操作这四种对象

  • blob对象
  • tree对象
  • commit对象
  • tag对象

所有对象都有唯一的index，git操作文件与其不如说是操作这些index，git将文件通过这些index巧妙简单的组织起来实现了各种功能。

总结

git是一个极其复杂的软件，但核心设计思想是很简单的；只有了解了这些设计的理念才能在平时的git操作中遇到任何问题都有解决的思路。
